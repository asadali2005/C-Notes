2.1 Introduction to Functions

A function is a named, reusable sequence of statements designed to perform a particular job. Think of it as a mini-program inside your program — you define the instructions once and can invoke them as many times as you like.

📖 Definition — Function
A function is a reusable block of code that performs a well-defined task. It has a name, may accept inputs (parameters), executes statements, and may return a result.

C++ recognises two broad categories of functions:

TypeWho writes it?Examples
User-defined functionsYou, the programmeradd(), readNumber()
Standard library functionsThe C++ Standard Librarystd::cout, std::sqrt()
✅ Key Rule
Every executable C++ program must contain a function named main(). This is the entry point — the first function the operating system calls when the program runs.

Function Execution & Nomenclature

When the CPU encounters a function call, it pauses the currently-running function, remembers where it stopped (like a "bookmark"), jumps to the called function, executes it fully, and then resumes from the bookmark.

Function Syntax

To call a function, write its name followed by parentheses and a semicolon:

⚠️ Important — No Nested Functions
C++ does NOT allow you to define one function inside another function's body. Every function must be defined at the top level (outside all other functions). Attempting to nest a function definition will cause a compiler error.

Functions can be called multiple times, and functions can call other functions. This nesting of calls (not definitions) is perfectly legal and central to how programs are structured.

2.1 Quiz & Solutions

📝 Quiz 2.1
Q1. In a function definition, what are the curly braces and the statements in-between called?
✔ Answer
The function body. It contains all the statements that execute when the function is invoked.
Q2. What does the following program print?
quiz_2_1.cpp 
#include <iostream>

void doB() { std::cout << "In doB()\n"; }

void doA() {
    std::cout << "In doA()\n";
    doB();
}

int main() {

    doB();
    std::cout << "Ending main()\n";

✔ Answer
Trace the execution step-by-step:

1. main() starts → prints "Starting main()"
2. doA() called → prints "In doA()", then calls doB() → prints "In doB()"
3. Back in main(), doB() called again → prints "In doB()"
4. main() ends → prints "Ending main()"

In doA()
In doB()
In doB()
Ending main()

📌 Section 2.1 Summary

  • Functions are reusable named blocks of code.
  • main() is the required entry point for every C++ program.
  • Functions execute top-to-bottom; the caller is paused and bookmarked while the callee runs.
  • Key terms: function call, caller, callee, invocation.
  • Nested function definitions are illegal in C++. Nested function calls are perfectly fine.

2.2 Function Return Values

Many functions need to pass a result back to whoever called them. This is done with a return value. Two things must be set up:

  1. Return Type: Stated in the function header — tells the compiler (and the reader) what kind of value will come back.
  2. Return Statement: Written inside the function body — specifies the actual value to send back.
📐 Return Statement Syntax
return expression;

The expression is evaluated, a copy of the result is handed to the caller, and the function immediately exits.
return_value.cpp 
#include <iostream>

int returnFive()
{
    return 5;   // Returns the integer value 5 to the caller
}

int main()
{
    std::cout << returnFive() << '\n';  // Prints: 5

    int x { returnFive() };           // x is initialized to 5

    returnFive();                       // Value returned but discarded — legal, just wasteful

    return 0;
}
💡 Return by Value — Key Mechanism
When you write return x + y;, the expression x + y is evaluated, and a copy of the result is handed back to the caller. The original variables inside the function are unaffected and then destroyed when the function exits.

main() and Status Codes

main() is special: it must return an int. This integer is called a status code (or exit code) and is passed to the operating system to indicate whether the program succeeded or failed.

Return ValueMacroMeaning
0EXIT_SUCCESSProgram ran successfully
Any non-zero valueEXIT_FAILUREProgram encountered an error
💡 Macros in <cstdlib>
EXIT_SUCCESS and EXIT_FAILURE are defined in the <cstdlib> header. Using them makes your code's intent immediately clear to other programmers.
⚠️ Explicit calls to main() are illegal
You cannot call main() from within your own code. The C++ standard explicitly forbids it.

Undefined Behavior & DRY Principle

⚠️ Critical Rule — Value-Returning Functions Must Return
Any function with a non-void return type must return a value. If execution reaches the end of such a function without hitting a return statement, the result is undefined behavior — the program may crash, produce garbage, or behave unpredictably.

Exception: main() implicitly returns 0 if no return is present — but explicitly writing return 0; is best practice.

A function can only return a single value. If you need multiple values, use other techniques (e.g., structs, references, or output parameters — covered in later chapters).

📖 DRY Principle — Don't Repeat Yourself
Code or logic that needs to be done multiple times should be written once inside a function and called wherever needed. Benefits: fewer bugs, easier updates, shorter code, better readability.

2.2 Quiz & Solutions

📝 Quiz 2.2
Q1a. What does cout << return7() + return9(); print, given return7() returns 7 and return9() returns 9?
✔ Answer
16. Both functions are called, their return values (7 and 9) are added, and the result is printed.
Q1b. return9() is defined inside return7(). What happens?
✔ Answer
Compiler error. Nested function definitions are illegal in C++.
Q1c. return7(); return9(); appear in main() with nothing printed. What happens?
✔ Answer
Compiles and runs successfully, outputs nothing. The return values are simply discarded. Ignoring a return value is legal (though sometimes a logical error in real programs).
Q1d. A function getNumbers() contains return 5; followed by return 7;. Its return value is printed twice. What prints?
✔ Answer
5 both times. The function returns at the first return statement it encounters. The second return 7; is unreachable code and never executes.
Q1e. int return 5() { return 5; }. What happens?
✔ Answer
Compiler error. return 5 is not a valid function identifier. Identifiers cannot contain spaces or start with a keyword like return.
Q2. What does "DRY" stand for, and why is it a useful practice?
✔ Answer
"Don't Repeat Yourself." When logic is written once and called many times: (1) updates only need to be made in one place, (2) the chance of introducing inconsistencies or typos is minimized, (3) code is shorter and easier to read.

📌 Section 2.2 Summary

  • Value-returning functions declare their return type and use return expr; to send a copy of a value back.
  • main() returns int; 0 = success, non-zero = failure.
  • Failing to return a value from a non-void function = undefined behavior.
  • Functions return a single value. The return value can be used or discarded.
  • Follow the DRY Principle: extract repeated logic into functions.

2.3 Void Functions

Not every function needs to return a value. If a function's only purpose is to perform an action (like printing output), it should have a void return type.

void_functions.cpp 
#include <iostream>

void printHello()
{
    std::cout << "Hello!\n";
    // No return statement needed — function exits automatically here
}

void earlyExit(bool flag)
{
    if (flag)
        return;   // Early return — exits the function immediately
    std::cout << "Flag was false\n";
}
RuleDetail
No return statement requiredA void function automatically returns at the closing }
Empty return; is allowedUsed for early exit; redundant at end of function
Cannot return a valuereturn 5; in a void function = compiler error
Cannot be used in expressions needing a valuestd::cout << printHello(); = compiler error
⚠️ Common Mistake
Trying to print or assign the result of a void function is a compiler error:
std::cout << printHello(); // ERROR — void has no value to send to cout

2.3 Quiz & Solutions

📝 Quiz 2.3
Q1a. printA() prints "A", printB() prints "B". Both are void. What does printA(); printB(); output?
✔ Answer
A and B on separate lines. Both functions execute and print their output sequentially.
Q1b. What about std::cout << printA() << '\n';?
✔ Answer
Compiler error. printA() returns void, which has no value. Attempting to pass it to std::cout via << is illegal.

📌 Section 2.3 Summary

  • Use void when a function has no value to return.
  • Void functions automatically exit at the end of their body.
  • An empty return; causes an early exit from a void function.
  • You cannot return a value from a void function, and you cannot use a void function call inside an expression that expects a value.

2.4 Function Parameters & Arguments

Functions often need external data to work with. Parameters and arguments are the mechanism for passing data into a function.

Parameter

A variable declared in the function header. It acts as a placeholder that the function will use internally. Defined once in the function signature.

int add(int x, int y)
x and y are parameters

Argument

The actual value supplied by the caller when the function is invoked. It is copied into the corresponding parameter.

add(4, 5)
4 and 5 are arguments

Pass by Value

📖 Pass by Value
When a function is called, each argument is evaluated, and a copy of its value is used to initialize the corresponding parameter. Changes to the parameter inside the function do not affect the original argument.
parameters.cpp 
#include <iostream>

int add(int x, int y)    // x and y are parameters
{
    return x + y;
}

int multiply(int a, int b)
{
    return a * b;
}

int main()
{
    std::cout << add(4, 5) << '\n';               // Literals as arguments → 9
    std::cout << add(1, multiply(2, 3)) << '\n';   // Function return as argument → 7
    return 0;
}

Arguments can be:

  • Literals: add(4, 5)
  • Variables: add(a, b)
  • Expressions: add(x + 1, y * 2)
  • Return values of other calls: add(1, multiply(2, 3))

Unnamed Parameters

If a parameter exists in the function signature but is not used in the body, you can omit its name to suppress compiler warnings. The type is still required:

unnamed_param.cpp 
void doSomething(int /*count*/)
{
    // 'count' is not needed here — name omitted, type kept
    // Commenting the name (/*count*/) documents what it represents
}

2.4 Quiz & Solutions

📝 Quiz 2.4
Q1. What is wrong with void multiply(int x, int y) { return x * y; }?
✔ Answer
The function is declared as void (returns nothing) but the return x * y; statement tries to return an int. This is a compiler error. The return type should be int.
Q2. What is wrong with the following?
int multiply(int x, int y) { int product { x * y }; } called as multiply(4)
✔ Answer
Two errors:
1. multiply() is a value-returning function (returns int) but has no return statement → undefined behavior.
2. The call multiply(4) provides only 1 argument for a function requiring 2 parameters → compiler error.
Q3. What does multiply(add(1, 2, 3), 4) print (assuming add takes 3 ints and returns their sum)?
✔ Answer
add(1, 2, 3) = 6. Then multiply(6, 4) = 24. Prints 24.
Q4 & Q5. Write a function doubleNumber() and a complete program to use it.
✔ Answer
double_number.cpp 
#include <iostream>

int doubleNumber(int x)
{
    return x * 2;
}

int main()
{
    std::cout << "Enter an integer: ";
    int input{};
    std::cin >> input;
    std::cout << "Double that number is: " << doubleNumber(input) << '\n';
    return 0;
}

📌 Section 2.4 Summary

  • Parameters are variables in the function header; arguments are values passed at the call site.
  • C++ uses pass by value: arguments are copied into parameters.
  • Arguments can be literals, variables, expressions, or function return values.
  • Unused parameters can have their name omitted (keep the type); commenting the name is good practice.

2.5 Local Scope

Variables defined inside a function body, and the function's parameters themselves, are collectively called local variables.

Lifetime vs. Scope — A Critical Distinction

Lifetime (Runtime)

The period between a variable's creation and destruction. Local variables are created when execution reaches their definition, and destroyed in reverse order at the end of the block {} containing them.

Scope (Compile-time)

Determines where in the source code an identifier can be accessed. Local variables have block scope: usable only from their point of definition to the closing } of their block.

💡 Functional Separation
Two different functions can each have a local variable named x without any conflict, because their scopes are completely separate. Each x refers only to the one in its own function.

Temporary Objects

📖 Temporary Object
An unnamed object created by the compiler to hold a value temporarily — for example, the copy of a value returned by a function before it's assigned to a variable. Temporaries are destroyed at the end of the full expression in which they appear.

Best Practices

  • Define local variables as close to their first use as possible.
  • Use a function parameter when the caller must supply the initial value.
  • Use a local variable (defined inside the function) for values computed internally.

2.5 Quiz & Solutions

📝 Quiz 2.5
Q1. What does the program print?
scope_quiz.cpp 
void doIt(int x)
{
    int y{ 4 };
    std::cout << "doIt: x = " << x << " y = " << y << '\n';
    x = 3;
    std::cout << "doIt: x = " << x << " y = " << y << '\n';
}

int main()
{
    int x{ 1 };
    int y{ 2 };
    std::cout << "main: x = " << x << " y = " << y << '\n';
    doIt(x);
    std::cout << "main: x = " << x << " y = " << y << '\n';
}
✔ Answer
Trace:
• main's x=1, y=2 → "main: x = 1 y = 2"
• doIt(x) called: x is copied as 1, y local to doIt = 4 → "doIt: x = 1 y = 4"
• Inside doIt, parameter x reassigned to 3 → "doIt: x = 3 y = 4"
• Back in main: x is still 1 (pass by value — unchanged!), y still 2 → "main: x = 1 y = 2"

Output:
main: x = 1 y = 2
doIt: x = 1 y = 4
doIt: x = 3 y = 4
main: x = 1 y = 2

📌 Section 2.5 Summary

  • Local variables and parameters exist only within the function/block that defines them.
  • Lifetime = when the variable exists in memory (runtime). Scope = where it can be accessed (compile-time).
  • Local variables have block scope — created at definition, destroyed at end of block.
  • Pass by value means changes to parameters do not affect original arguments.
  • Each function has its own isolated namespace for variable names.

2.6 Why Functions are Useful

Functions are the primary tool for managing complexity in C++ programs. Here is a structured breakdown of the benefits:

BenefitExplanation
OrganizationBreaks large, complex programs into small, manageable, named chunks.
ReusabilityWrite once, call anywhere — eliminates copy-paste duplication (DRY).
TestingSelf-contained functions are easy to test in isolation.
ExtensibilityUpdate the logic in one function, and all callers immediately benefit.
AbstractionCallers only need to know what a function does and its interface — not how it's implemented internally.

Design Guidelines

  • A function should perform one, and only one clearly-defined task.
  • If a function grows large or does multiple things, refactor it into smaller sub-functions.
  • Group any repeated sequence of statements into a named function.
  • Functions work best when their inputs and outputs are well-defined.
📖 Refactoring
The process of restructuring existing code — often by splitting a complex function into multiple smaller sub-functions — without changing its external behavior. The goal is improved readability and maintainability.

📌 Section 2.6 Summary

  • Functions provide organization, reusability, testability, extensibility, and abstraction.
  • Each function should do exactly one thing well.
  • Refactoring breaks large functions into smaller, more focused sub-functions.

2.7 Forward Declarations & Definitions

The C++ compiler reads source files top-to-bottom. If you call a function before the compiler has seen its definition, it will produce an "identifier not found" error.

❌ Problem — Ordering Issue

main() calls add(), but add() is defined further down in the file. The compiler errors out.

✅ Solutions

1. Reorder definitions (put add() above main()).
2. Use a forward declaration (function prototype).

📖 Forward Declaration
Tells the compiler about the existence and type signature of an identifier before its full definition appears. For functions, this is done with a function prototype: return type, name, parameter types, and a semicolon — no body.
forward_decl.cpp 
#include <iostream>

int add(int x, int y);   // Forward declaration (function prototype)
                            // Parameter names are optional but recommended

int main()
{
    std::cout << add(3, 4) << '\n';   // Compiler trusts the prototype above
    return 0;
}

int add(int x, int y)    // Actual definition — can appear after main()
{
    return x + y;
}

Declarations vs. Definitions

DeclarationDefinition
PurposeTells the compiler an identifier exists and its typeActually implements / instantiates the entity
Body?No (for functions)Yes
Allocates memory?NoYes (for variables)
Exampleint add(int, int);int add(int x, int y) { return x+y; }
RelationshipAll definitions are declarations. Not all declarations are definitions.

One Definition Rule (ODR)

📐 One Definition Rule — Three Parts
  1. Within a file: An entity (variable, function, type) may have only one definition per translation unit.
  2. Within a program: Normal functions and variables may have only one definition across all files. (Violation = linker error.)
  3. Special exception: Types, templates, and inline functions may have identical duplicate definitions in different files — this is required for headers to work.

2.7 Quiz & Solutions

📝 Quiz 2.7
Q1. What is a function prototype?
✔ Answer
A declaration of a function that includes its return type, name, and parameter types, terminated with a semicolon — but no function body.
Q2. What is a forward declaration?
✔ Answer
A statement that informs the compiler about an identifier's existence and type before the full definition appears in the code.
Q3. How do we declare a forward declaration for functions?
✔ Answer
Using a function prototype — the function signature (return type, name, parameters) followed by a semicolon.
Q4. Write the forward declaration for: int doMath(int first, int second, int third, int fourth) { ... }
✔ Answer
int doMath(int first, int second, int third, int fourth);
Q5. Compile/Link/Success — diagnose each scenario:
✔ Answer
CaseSituationResult
aCall add(3,4,5); prototype is int add(int,int)Compile Fails — argument count mismatch
bCall add(3,4,5); prototype is int add(int,int); definition is int add(int,int,int)Compile Fails — compiler only sees 2-param prototype

📌 Section 2.7 Summary

  • The compiler reads top-to-bottom; a function must be declared or defined before it is called.
  • A forward declaration (function prototype) tells the compiler a function exists without its body.
  • Declaration = announces existence. Definition = provides implementation. All definitions are declarations.
  • The One Definition Rule: entities may only be defined once per translation unit (and once per program for non-inline entities).

💡 How Multi-File Programs Work
The compiler compiles each .cpp file independently — it forgets about any other file. To call a function defined in add.cpp from main.cpp, you must provide a forward declaration in main.cpp. The linker then resolves the connection between the call and the definition.
⚠️ Never #include .cpp Files
Do NOT use #include "add.cpp". This causes the entire contents of add.cpp to be pasted into the current file, which leads to duplicate definition errors. Instead, add .cpp files to your IDE project or compilation command, and use header files for declarations.

2.9 Naming Collisions & Namespaces

📖 Naming Collision
Occurs when two identical identifiers are introduced into the same scope. The compiler (for local conflicts) or linker (for global conflicts) cannot determine which one to use and reports an error.
📖 Namespace
A named region of code that provides a separate scope for identifiers. Names declared inside a namespace are isolated from names in other scopes, preventing collisions.
Namespace ConceptDetails
Global namespaceAny identifier not inside a class, function, or named namespace automatically belongs here.
std namespaceAll C++ Standard Library identifiers (like cout, cin) live here to avoid collisions.
Namespace contentsCan contain declarations and definitions. No loose executable statements (only inside function bodies).

Accessing Namespace Members

✅ Explicit Scope Resolution (Best Practice)

std::cout — uses :: (scope resolution operator) to specify which namespace the identifier belongs to. Always safe, always clear.

❌ Using Directive (Avoid)

using namespace std; — imports all names from std into the current scope. Re-introduces the risk of naming collisions and reduces code clarity.
📐 Scope Resolution Operator
namespaceName::identifier

Example: std::cout means "the identifier cout from the std namespace."

📌 Section 2.9 Summary

  • Naming collisions occur when two identical names exist in the same scope.
  • Namespaces provide isolated scopes to prevent collisions.
  • The C++ Standard Library uses the std namespace.
  • Always use explicit scope resolution (std::cout). Avoid using namespace std;.

2.10 Introduction to the Preprocessor

Before your source code is compiled, it passes through the preprocessor — a text-manipulation program that acts on preprocessor directives. The output is called a translation unit, which the compiler then processes.

💡 Key Properties of Preprocessor Directives
  • Begin with the # symbol.
  • Do NOT end with a semicolon.
  • Are processed before compilation begins.
  • Are file-scoped: a #define in one .cpp file does not affect another file.

Common Preprocessor Directives

DirectivePurposeExample
#includeReplaces the directive with the full contents of the specified file#include <iostream>
#define (with text)Creates an object-like macro — every occurrence of the macro name is replaced with the text#define MY_NAME "Alex"
#define (without text)Creates a macro that is simply "defined" — used for conditional compilation#define USE_YEN
#ifdef / #ifndefConditional compilation: includes code if a macro is (or is not) defined#ifdef USE_YEN
#endifEnds an #ifdef / #ifndef / #if block#endif
#if 0 ... #endifPermanently excludes a code block — great for commenting out code with existing multi-line commentsSee below
preprocessor_examples.cpp 
#include <iostream>   // Pastes iostream's entire content here

#define MY_NAME "Alex"   // Object-like macro: MY_NAME → "Alex"
#define USE_YEN          // Macro with no value — used as a flag

#ifdef USE_YEN
    // This block compiles only if USE_YEN is defined
    std::cout << "Using Yen\n";
#endif

#ifndef USE_DOLLARS
    // This block compiles only if USE_DOLLARS is NOT defined
    std::cout << "Not using dollars\n";
#endif

#if 0
    // Entire block is excluded from compilation
    /* Even nested multi-line comments are fine here */
    std::cout << "This never runs\n";
#endif
⚠️ Modern C++ Advice on #define Macros
Text-substitution macros (e.g., #define MAX_SIZE 100) should be avoided in modern C++. They have no type safety and can cause confusing bugs. Prefer constexpr variables and const for constants instead. Flag-style macros (#define USE_YEN) are still commonly used for conditional compilation.

📌 Section 2.10 Summary

  • The preprocessor runs before the compiler, manipulating source text based on directives starting with #.
  • #include pastes file contents; #define creates macros; #ifdef/#ifndef/#endif enable conditional compilation.
  • #if 0 is an effective way to comment out code blocks.
  • Directives are file-scoped — they don't bleed into other .cpp files.
  • Prefer constexpr over text-substitution macros in modern C++.

Use header guards in every header (see 2.12)Prevents duplicate declaration errors
⚠️ Transitive Includes
If myheader.h includes <iostream>, and your .cpp includes myheader.h, then std::cout may work — but only because of transitive inclusion. This is fragile. If myheader.h later stops including <iostream>, your code breaks. Always explicitly include what you directly use.

📌 Section 2.11 Summary

  • Header files centralise forward declarations, avoiding repetition.
  • Use < > for standard library headers, " " for user-defined headers.
  • Headers should contain declarations only, not definitions (to avoid ODR violations).
  • Always explicitly include headers you use; never rely on transitive includes.
  • Pair every .cpp with its own .h and #include the header in the .cpp.

2.12 Header Guards

When multiple files include the same header (or a header is included indirectly multiple times), the compiler may see its declarations more than once in a single translation unit, causing errors.

📖 Header Guard (Include Guard)
A preprocessor technique that prevents the contents of a header file from being processed more than once in a single translation unit. Uses #ifndef, #define, and #endif.
square.h 
#ifndef SQUARE_H   // If SQUARE_H is NOT defined (first time including)
#define SQUARE_H   // Define SQUARE_H to mark it as "already included"

// All declarations go here
int getSquareSides();
int getSquarePerimeter(int);

#endif   // End of the guard block
// If SQUARE_H was already defined, the entire middle section is skipped

How the Guard Works — Step by Step

  1. First inclusion: SQUARE_H is not defined → preprocessor enters the block → defines SQUARE_H → processes all declarations.
  2. Second inclusion: SQUARE_H is now defined → the #ifndef condition is false → the entire block is skipped.
💡 Naming Convention
The macro name should be unique to avoid accidental collision with other guards. Convention: use the filename in ALL_CAPS with underscores, replacing the dot with _H (e.g., my_header.h → MY_HEADER_H).

#pragma once — Modern Alternative

Traditional Header Guard

Verbose, requires unique macro name, universally supported by all compilers including very old ones.

#pragma once

Single line, simpler, supported by all major modern compilers. Serves exactly the same purpose. Preferred in modern codebases.

square.h (with pragma once) 
#pragma once   // Modern alternative to traditional header guards

int getSquareSides();
int getSquarePerimeter(int);
⚠️ Important Limitation of Header Guards
Header guards prevent a header from being processed more than once within the same .cpp file (translation unit). They do NOT prevent the header from being included in different .cpp files — which is expected and necessary. Each .cpp file gets its own fresh copy of the preprocessor environment.

📌 Section 2.12 Summary

  • Header guards prevent a header's contents from being processed more than once per translation unit.
  • Mechanism: #ifndef MACRO_H / #define MACRO_H / content / #endif.
  • #pragma once is a simpler modern alternative supported by all major compilers.
  • Guards operate per translation unit — the same header can and should be included in multiple .cpp files.

2.13 How to Design Your First Programs

Writing code before thinking about design often leads to wasted effort, buggy software, and hard-to-maintain codebases. A structured design approach saves time overall.

High-Level Design Steps

StepAction
1. Define your goalExpress what the program should achieve in clear, user-facing terms (one or two sentences).
2. Define requirementsList the capabilities the program must have and any constraints (e.g., must not crash on bad input, must run on Windows).
3. Define tools & targetsChoose your IDE, OS, version control (e.g., Git), and backup strategy.
4. Break down the problemUse top-down or bottom-up decomposition to split the task into sub-tasks. Each sub-task becomes a candidate function.
5. Sequence eventsDetermine the logical order of operations (what must happen before what?).

Implementation Steps

  1. Outline main() with comments or stub functions (functions with empty or placeholder bodies) representing the overall flow.
  2. Implement one function at a time: write the prototype → write the body → test it in isolation.
  3. Final program test: test the whole program together once individual pieces are verified.
💡 Golden Advice — Start Simple
  • Start with the simplest version that works.
  • Add features incrementally, testing after each addition.
  • If something breaks, you know exactly what changed.
  • Optimise for correctness and maintainability before performance.
  • Avoid the temptation to write the entire program at once without testing.

📌 Section 2.13 Summary

  • Design before coding: goal → requirements → tools → decomposition → sequencing.
  • Decompose into sub-tasks which map naturally to functions.
  • Implement and test one function at a time.
  • Start simple, add features incrementally, test frequently.

📋 Master Summary & Revision Sheet — Chapter 2

Everything you need to know for a complete review of Chapter 2 in one place.

2.1 — Functions Basics

  • Reusable named blocks of code
  • main() = mandatory entry point
  • CPU bookmarks caller, runs callee, returns
  • No nested definitions (but nested calls are fine)
  • Syntax: returnType name() { body }

2.2 — Return Values

  • Non-void functions MUST return a value
  • Return sends a copy of the value
  • main() returns int (0 = success)
  • Missing return → undefined behavior
  • DRY: Don't Repeat Yourself

2.3 — Void Functions

  • No return value → use void
  • Auto-returns at closing brace
  • Empty return; = early exit
  • Cannot use in expressions needing a value

2.4 — Parameters & Arguments

  • Parameter = variable in header
  • Argument = value at call site
  • Pass by value = arguments are copied
  • Can pass literals, variables, expressions
  • Unused params: omit name, keep type

2.5 — Local Scope

  • Local vars: block scope only
  • Lifetime (runtime) ≠ Scope (compile-time)
  • Destroyed at end of their block
  • Same-named vars in different functions = no conflict

2.6 — Why Functions

  • Organization, Reusability, Testing
  • Extensibility, Abstraction
  • One function = one task
  • Refactoring: split large functions

2.7 — Forward Declarations

  • Compiler reads top-to-bottom
  • Prototype = declaration without body
  • ODR: one definition per entity per program
  • All definitions are declarations

2.8 — Multiple Files

  • Each .cpp compiled independently
  • Linker connects object files
  • Use forward declarations across files
  • Never #include .cpp files!

2.9 — Namespaces

  • Naming collision = two same identifiers in scope
  • Namespaces isolate identifier scopes
  • std:: namespace for standard library
  • Always use std::cout, avoid using namespace

2.10 — Preprocessor

  • Runs before compilation, manipulates text
  • #include pastes file contents
  • #define creates macros
  • #ifdef / #ifndef / #endif = conditional
  • #if 0 = easy block comment-out

2.11 — Header Files

  • Centralize forward declarations in .h files
  • < > for standard, " " for user headers
  • Declarations only — no definitions
  • No transitive include reliance

2.12 — Header Guards

  • Prevent double-inclusion in same TU
  • #ifndef / #define / #endif pattern
  • #pragma once = modern shorthand
  • Don't prevent cross-.cpp inclusion (that's fine)

⚡ Critical Rules — Never Forget

  • Value-returning functions must return a value or it's undefined behavior.
  • No nested function definitions in C++.
  • Explicit calls to main() are illegal.
  • Never #include .cpp files.
  • Never use using namespace std; in production code.
  • Always use header guards in every header file.
  • Headers contain declarations only (not definitions) to avoid ODR violations.
  • Always explicitly include all headers you use — don't rely on transitive includes.
  • Pass by value means changes to parameters do not affect the caller's arguments.

🔗 The Complete Multi-File Structure (Quick Reference)

  • foo.h — Contains declarations + header guards. Included by anyone who needs the functions.
  • foo.cpp — Contains definitions. Includes foo.h to verify its own declarations.
  • main.cpp — Includes foo.h to get declarations. Linker connects calls to definitions in foo.cpp.
