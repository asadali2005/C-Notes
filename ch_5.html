Section 5.1

Constant Variables (Named Constants)

In programming, a constant is a value that cannot be changed after it is set during a program's execution. C++ gives us two broad categories of constants:

CategoryDescriptionExample
Named constantsA constant value bound to an identifier (symbolic constant)const int maxScore { 100 };
Literal constantsA raw constant value with no identifier5, 1.2, 'a'

There are three ways to define a named constant in C++:

  1. Constant variables (using const or constexpr)
  2. Object-like macros with substitution text (using #define) — not recommended
  3. Enumerated constants (enum) — covered in a later chapter

Declaring a const Variable

To make a variable constant, place the const keyword adjacent to the type. The compiler will prevent any reassignment after initialization.

const_declaration.cpp 
const double gravity { 9.8 };  // ✅ Preferred: const before the type
int const sidesInSquare { 4 }; // ✅ Acceptable "East const" style, but not preferred

// gravity = 9.9; // ❌ ERROR: Cannot assign to a const variable
✅ Best Practice
Always place const before the type. This is the universally preferred style in modern C++.

Initialization Rules

⚠️ Warning — Must Initialize!
A const variable must be initialized when it is defined. You cannot define it first and assign later, because that would count as a reassignment.

The initializer can be:

  • A literal: const int x { 5 };
  • Another variable: const int y { x };
  • A runtime value (e.g., user input) — the initializer does NOT have to be a compile-time value
const_init.cpp 
int age;
std::cin >> age;                      // Runtime value

const int constAge { age };          // ✅ Valid: initialized with a runtime value
// constAge = 6;                       // ❌ ERROR: Cannot reassign after init

Naming Conventions

Because const variables behave like regular variables in every way except reassignment, you should follow the same naming convention as normal variables — typically camelCase.

  • ✅ const int maxStudents { 30 };
  • ❌ const int MAX_STUDENTS { 30 }; — ALL_CAPS belongs to C-style macro naming, which we want to avoid in modern C++.

Const Function Parameters & Return Values

❌ Don't do this

  • Passing by value with const adds noise with no real benefit — the copy is destroyed at end of function anyway
  • Returning a const value by value prevents the compiler from using move semantics optimizations

✅ Do this instead

  • Omit const for pass-by-value parameters
  • Omit const on return-by-value
  • Do use const for pass-by-reference parameters (covered later)

Why Use Constant Variables?

💡 Key Benefits
  • Bug prevention: The compiler catches any accidental reassignment.
  • Compiler optimization: The compiler can make aggressive speed/memory optimizations knowing the value will never change.
  • Reduced mental load: You don't need to track if/when a value changes while debugging.
  • Self-documenting: Signals intent to other programmers ("this value should not change").
✅ Best Practice
Make every variable const by default. Only remove const if you specifically need to modify it.

Prefer const Over Preprocessor Macros (#define)

In C-style code, programmers used #define to create symbolic constants:

define_example.cpp 
#define MY_NAME "Alex"          // ❌ Bad C-style practice
const std::string myName { "Alex" }; // ✅ Preferred C++ style
⚠️ Why #define is Bad
  1. No scope: Macros ignore C++ scoping rules — they replace text everywhere in the file after the #define line.
  2. Debugger invisible: The debugger sees the substituted value, not the macro name, making debugging cryptic.
  3. Not typed: Macros have no type information, so the compiler cannot type-check them.
  4. Bizarre errors: If a macro name collides with something else, the resulting errors are very hard to interpret.

📋 Section 5.1 Summary

  • A constant variable uses const and cannot be reassigned after initialization.
  • Always initialize const variables at the point of definition.
  • Place const before the type; use camelCase naming.
  • Don't use const for pass-by-value params or return-by-value.
  • Prefer const/constexpr over #define macros — always.
Section 5.2

Literals

📖 Definition — Literal
A literal (also called a literal constant) is a fixed value written directly in source code. The compiler determines the type of a literal from its form. Literals have no memory address — they exist as immediate values baked into the compiled program.
LiteralExampleDefault Type
Integer5, -3, 0int
Booleantrue, falsebool
Floating-point1.2, 3.14double (NOT float!)
Character'a', 'Z'char
C-style string"Hello"const char[]

Literal Suffixes

You can override the default type of a literal by appending a suffix:

SuffixTypeExampleNotes
u or Uunsigned int5u
l or Llong5LUse uppercase L — lowercase looks like 1
ll or LLlong long5LL
ul or ULunsigned long5UL
ull or ULLunsigned long long5ULL
f or Ffloat5.0fVery common — use to avoid double→float warning
sstd::string"Hello"sRequires using namespace std::string_literals;
svstd::string_view"Hello"svRequires using namespace std::string_view_literals;
✅ Best Practice
Prefer uppercase L over lowercase l (which is easily confused with the digit 1 in many fonts).

Floating-Point Literals & the Double Default

⚠️ Common Mistake — float vs double Literal
Floating-point literals are double by default, not float.
float_warning.cpp 
float f { 4.1 };   // ⚠️ 4.1 is a double — narrowing conversion to float, loses precision
float g { 4.1f };  // ✅ 4.1f is a float literal — no conversion needed

Floating-point literals also support scientific notation:

scientific.cpp 
double avogadro { 6.02e23 };   // 6.02 × 10²³
double electron { 1.6e-19 };   // 1.6 × 10⁻¹⁹

String Literals & the Null Terminator

📖 Definition — C-Style String Literal
A sequence of characters enclosed in double quotes " ". Its type is const char[]. It contains a hidden null terminator character (\0) at the end.
💡 Important — Null Terminator
The string "Hello" is 6 characters long, not 5:
'H' 'e' 'l' 'l' 'o' '\0'

C-style string literals are const objects that exist for the entire lifetime of the program.

Magic Numbers

📖 Definition — Magic Number
A magic number is a hard-coded literal value whose meaning is not immediately obvious from context, or whose value might need to change later.
magic_numbers.cpp 
// ❌ BAD: What does 30 mean? What if it needs to change?
int totalScore { numStudents * 30 };

// ✅ GOOD: Intent is clear, easy to update in one place
constexpr int maxPointsPerStudent { 30 };
int totalScore { numStudents * maxPointsPerStudent };
✅ Best Practice
Avoid magic numbers. Extract every meaningful literal into a named constexpr symbolic constant.

📋 Section 5.2 Summary

  • Floating-point literals are double by default — use the f suffix for float.
  • Use suffixes to make literal types explicit (e.g., 5U, 5LL, 4.1f).
  • C-style string literals include a hidden null terminator \0 and live for the whole program.
  • Avoid magic numbers — replace with named symbolic constants.
Section 5.3

Numeral Systems

Computers operate in binary (base-2), but we often need to work with other numeral systems. C++ lets you write integer literals in four bases:

SystemBaseDigits UsedC++ PrefixExampleNotes
Decimal100–9Noneint x { 12 };Default; most common
Octal80–70 (zero)int x { 014 };Highly discouraged — confusing
Hexadecimal160–9, A–F0xint x { 0xC };Common for memory/bitwise work
Binary20, 10bint x { 0b1100 };C++14 and later
💡 Why Hexadecimal?
One hexadecimal digit represents exactly 4 bits (a nibble). Two hex digits represent one byte (8 bits). This makes hex extremely compact and readable for memory addresses and binary data:
  • 0xFF = 0b1111'1111 = 255 (1 byte, all bits set)
  • 0x1A = 0b0001'1010 = 26

Digit Separators (C++14)

Starting in C++14, you can use a single quote ' as a visual separator inside numeric literals. It has no effect on the value — it is purely cosmetic.

separators.cpp 
int  bin   { 0b1011'0010 };       // Binary: groups of 4 bits
long value { 2'132'673'462 };     // Decimal: thousands separator
int  hex   { 0xFF'EC'D1'09 };     // Hex: byte grouping

Outputting Values in Different Bases

std::cout outputs in decimal by default. Use I/O manipulators to change the output base:

output_bases.cpp 
#include <iostream>
#include <bitset>   // Needed for binary output

int main() {
    int x { 12 };

    std::cout << std::hex << x << '\n';     // Outputs: c
    std::cout << std::oct << x << '\n';     // Outputs: 14
    std::cout << std::dec << x << '\n';     // Outputs: 12 (back to decimal)

    // For binary output, use std::bitset<N> where N = number of bits to display
    std::cout << std::bitset<8>{ x } << '\n';  // Outputs: 00001100

    return 0;
}
💡 I/O Manipulators are Sticky
Once you set std::hex or std::oct, ALL subsequent integer outputs use that base until you explicitly set std::dec to restore decimal. Don't forget to reset!

📝 Quiz 5.3 — Worked Solutions

Question 1: What is decimal 32 in binary and hexadecimal?

✅ Solution

Step 1 — Hexadecimal:
32 ÷ 16 = 2 remainder 0. So: 2 × 16¹ + 0 × 16⁰ = 0x20

Step 2 — Binary:
32 = 2⁵. So it's a 1 followed by five 0s: 0b10'0000 (= 0b100000)
You can verify: 0b10'0000 = 1×32 + 0+0+0+0+0 = 32 ✓

Cross-check: Hex 0x20 = 2×16 + 0 = 32 ✓  |  Binary 100000 = 32 ✓

📋 Section 5.3 Summary

  • C++ supports decimal (no prefix), octal (0), hex (0x), and binary (0b, C++14).
  • Avoid octal — it is confusing and rarely needed.
  • Use digit separators ' for readability (C++14+).
  • Use std::hex/std::oct/std::dec to control output; use std::bitset<N> for binary.
Section 5.4

The As-If Rule & Compile-Time Optimization

📖 Definition — Optimization
Optimization is the process of modifying a program so that it runs faster, uses less memory, or both — without changing what the program does from the user's perspective.

The As-If Rule

📐 The As-If Rule
The C++ compiler is permitted to transform or reorder your code in any way it likes — as long as the observable behavior of the program remains the same. The compiler doesn't need to ask permission or tell you what it changed.

This rule gives the compiler enormous freedom to optimize. "Observable behavior" is defined narrowly as things the user can actually see: output to the screen, file writes, etc. Internal variable names, intermediate calculations, and the order of non-observable operations are all fair game for the compiler to change.

Compile-Time Evaluation Techniques

Evaluating computations at compile-time means the executable doesn't have to do that work at runtime, resulting in smaller and faster programs. The three key techniques are:

TechniqueWhat it doesExample (before → after)
Constant Folding Evaluates expressions with only literal operands at compile-time int x { 3 + 4 }; → int x { 7 };
Constant Propagation Replaces variables whose values are known constants with their literal values int x { 7 }; cout << x; → cout << 7;
Dead Code Elimination Removes code that executes but has no observable effect A variable that is initialized but never read is removed entirely
optimization_example.cpp 
// What you write:
int x { 3 + 4 };              // Constant folding: 3+4 computed at compile-time
std::cout << x;               // Constant propagation: x replaced with 7

// What the compiler effectively generates (conceptually):
std::cout << 7;               // x might not even exist in the final binary!
💡 Connection to const
Declaring variables as const gives the compiler a guarantee that the value will never change. This enables the compiler to safely apply constant propagation even if the variable's value is used in many places. Without const, the compiler must be conservative and cannot propagate unless it can prove the value never changes.
⚠️ Debug vs Release Builds
Debug builds disable most optimizations so that variables remain in memory and the debugger can inspect them. In release/optimized builds, variables may be completely eliminated, making debugging extremely difficult. This is expected and normal behavior.

📋 Section 5.4 Summary

  • The As-If Rule: The compiler may transform code freely as long as observable behavior is unchanged.
  • Constant Folding: Literal-only expressions computed at compile-time.
  • Constant Propagation: Known-value variables replaced with their literal values.
  • Dead Code Elimination: Unused variables/code removed from the binary.
  • const enables more aggressive optimizations.
Section 5.5

Constant Expressions

📖 Definition — Constant Expression
A constant expression is a non-empty sequence of literals, constant variables, operators, and function calls that the compiler is capable of evaluating entirely at compile-time. Every element in the expression must itself be a compile-time constant.

Distinguishing runtime from compile-time evaluation is a fundamental C++ concept:

✅ Compile-Time Expression

  • All values are known when code is compiled
  • Can be used as array sizes, template arguments, etc.
  • Results in faster, smaller executables
  • Bugs caught at compile-time, not runtime

⏱ Runtime Expression

  • Values not known until the program runs
  • Cannot be used as array sizes or template args
  • Flexible — can involve user input, files, etc.

What CAN Be in a Constant Expression?

  • Literal values: 5, 1.2, true, 'a', "Hello"
  • Operators applied to constant expression operands: 3 + 4, 2 * 5
  • const integral variables initialized with a constant expression: const int x { 5 };
  • constexpr variables (any type)
  • Calls to constexpr functions

What CANNOT Be in a Constant Expression?

  • Non-const variables (even if they happen to hold a constant-like value)
  • const non-integral types (e.g., const double, const float) — use constexpr for those
  • Runtime operations: std::cin, std::cout, function calls that execute at runtime
  • Expressions involving any of the above
💡 Key Insight — Required vs. Optional
The compiler is only required to evaluate a constant expression at compile-time in contexts that demand a compile-time value (e.g., array sizes, constexpr variable initializers, template arguments). In other contexts, it may choose to evaluate at compile-time via the as-if rule — but is not obligated to.

📝 Quiz 5.5 — Worked Solutions

Question 1: For each variable, identify if (a) the initializer is a constant expression, and (b) the variable itself is a constant expression.

✅ Detailed Solutions
VariableInitializer CE?Variable CE?Reason
char a { 'q' }; ✅ Yes ❌ No 'q' is a literal (CE). But a is not const, so it's not a CE.
const int b { 0 }; ✅ Yes ✅ Yes 0 is a literal. b is const integral initialized with CE → it IS a CE.
const double c { 5.0 }; ✅ Yes ❌ No 5.0 is a literal. But double is non-integral — const is not enough; need constexpr.
const int d { a * 2 }; ❌ No ❌ No a is not a CE (not const), so a * 2 is not CE. Therefore d is not CE.
int e { c + 1.0 }; ❌ No ❌ No c is not a CE (const double, non-integral). So c + 1.0 is not CE.
const int f { d * 2 }; (if d were CE) ✅ Yes (hypothetical) ✅ Yes (hypothetical) If d is CE, then d * 2 is CE, and f (const integral + CE init) is CE.
const int g { getNumber() }; ❌ No ❌ No getNumber() is a regular runtime function — not a CE.
const int h {}; ✅ Yes ✅ Yes Value-initialization → initializes to 0, which is a literal CE. h is const integral + CE init → CE.

📋 Section 5.5 Summary

  • A constant expression can be fully evaluated at compile-time.
  • const integral variables initialized with a CE are themselves CEs.
  • const non-integral (e.g., double) variables are NOT CEs — use constexpr.
  • Non-const variables are never CEs, even if their value is a literal.
Section 5.6

Constexpr Variables

const has two significant limitations:

  1. It does not guarantee compile-time evaluation — only that the value won't change.
  2. A const variable of non-integral type (like double) is NOT a constant expression.

To solve both problems, C++11 introduced the constexpr keyword.

📖 Definition — constexpr
constexpr (short for "constant expression") declares a variable that is a guaranteed compile-time constant. Its initializer must be a constant expression. Works for any type, including double, float, etc.
constexpr_examples.cpp 
constexpr double gravity     { 9.8 };    // ✅ Works on non-integral types!
constexpr int    sum         { 4 + 5 };  // ✅ Expression evaluated at compile-time
constexpr int    maxStudents { 30 };   // ✅ Named constant for magic number

int age;
std::cin >> age;
// constexpr int myAge { age }; // ❌ ERROR: age is a runtime variable

const vs constexpr — Complete Comparison

Propertyconstconstexpr
Can be reassigned?❌ No❌ No
Guarantees compile-time eval?❌ No (may be runtime)✅ Yes (always compile-time)
Works for non-integral types (double)?❌ Not as CE✅ Yes
Initializer must be a CE?❌ No (can be runtime)✅ Yes (mandatory)
Implicitly const?It IS const✅ Yes (constexpr implies const)
Use caseRuntime-determined values that shouldn't changeCompile-time known values
✅ Best Practice — Which to Use
  • Use constexpr when the initializer is known at compile-time (literals, other constexpr values, etc.)
  • Use const when the initializer is only known at runtime (e.g., user input, file reads)
  • Never use a plain non-const, non-constexpr variable when the value doesn't need to change

📋 Section 5.6 Summary

  • constexpr guarantees compile-time evaluation and works for all types.
  • A constexpr variable is always const, but a const variable is not necessarily constexpr.
  • Use constexpr by default for constants; fall back to const only for runtime-determined constants.
Section 5.7

Introduction to std::string

C-style strings (e.g., "Hello") are primitive and difficult to work with — you cannot concatenate them with +, get their length easily, or resize them. Modern C++ provides std::string from the <string> header.

📖 Definition — std::string
std::string is a class in the C++ Standard Library that manages a dynamically-allocated sequence of characters. It is an owner of its data — it makes a copy of whatever string it is initialized with and manages its own memory automatically.
string_basics.cpp 
#include <iostream>
#include <string>

int main() {
    std::string name { "Alex" };   // Initialize
    name = "Jason";              // ✅ Can be reassigned (unlike const)
    std::cout << name << '\n';   // Outputs: Jason
    return 0;
}

std::string is powerful: it automatically expands to fit any length string. However, this dynamic memory allocation makes it relatively slow to initialize and copy.

String Input & Handling Whitespace

⚠️ Common Mistake — std::cin >> with Strings
std::cin >> name stops reading at the first whitespace character (space, tab, newline). So if the user types John Smith, only John is stored.

Use std::getline() to read an entire line including spaces:

getline_example.cpp 
#include <iostream>
#include <string>

int main() {
    std::string name{};
    
    // ✅ Correct way to read a full line with possible spaces
    // std::ws skips leading whitespace (including leftover newlines)
    std::getline(std::cin >> std::ws, name);
    
    std::cout << "You entered: " << name << '\n';
    return 0;
}
💡 Why std::ws is Essential
When you use std::cin >> someInt, the user presses Enter, which leaves a '\n' character sitting in the input buffer. The next call to std::getline will immediately read that leftover '\n' and return an empty string — seemingly skipping the input entirely. std::ws discards all leading whitespace (including that newline) before getline reads.

Rule: Always write std::getline(std::cin >> std::ws, yourString);

String Length

string_length.cpp 
std::string name { "Alex" };

// name.length() returns size_t (unsigned) — cast to avoid signed/unsigned mismatch
int len { static_cast<int>(name.length()) };  // ✅ Preferred

// C++20 alternative: std::ssize() returns a signed integer natively
int len2 { static_cast<int>(std::ssize(name)) };
⚠️ Unsigned Length Warning
name.length() (and name.size()) return size_t, which is an unsigned integer type. Comparing or doing arithmetic with signed integers can cause subtle bugs (e.g., if length were somehow 0 and you subtract 1, you get a huge positive number due to unsigned wraparound). Always static_cast<int> when using with signed arithmetic.

Passing and Returning std::string

❌ Never pass by value

Passing std::string by value creates an expensive copy of the entire string. Avoid this.

void print(std::string s) { ... } // BAD

✅ Returning by value is fine

When returning a local std::string, the compiler applies move semantics or NRVO (Named Return Value Optimization) — so no expensive copy occurs.

std::string getName() { return name; } // OK

std::string Literals (The s Suffix)

string_literals.cpp 
#include <string>

using namespace std::string_literals;  // Enables the 's' suffix

auto s1 = "Hello";   // type: const char[]  (C-style)
auto s2 = "Hello"s;  // type: std::string   (C++ style)

📝 Quiz 5.7 — Worked Solution

Question: Ask for the user's full name and age. Print the sum of their age and the number of characters in their name (count spaces).

✅ Full Solution
quiz_5_7.cpp 
#include <iostream>
#include <string>

int main() {
    std::cout << "Enter your full name: ";
    std::string name{};
    std::getline(std::cin >> std::ws, name);  // Reads full line with spaces

    std::cout << "Enter your age: ";
    int age{};
    std::cin >> age;

    int nameLength { static_cast<int>(name.length()) };

    std::cout << "Age + name length = "
              << (age + nameLength)
              << '\n';

    return 0;
}

📋 Section 5.7 Summary

  • std::string owns its data and dynamically resizes — powerful but costly to copy.
  • Use std::getline(std::cin >> std::ws, name) for full-line input.
  • Cast .length() to int using static_cast<int> to avoid signed/unsigned issues.
  • Never pass std::string by value; returning by value is OK.
  • Use the s suffix with using namespace std::string_literals; for std::string literals.
Sections 5.8–5.9

Introduction to std::string_view

Because std::string always makes a copy of its data, passing it by value is expensive. C++17 introduced std::string_view (from <string_view>) as a lightweight, read-only window into existing string data.

📖 Definition — std::string_view
std::string_view is a non-owning, read-only view of a string. It stores only a pointer to the start of the string and its length — it holds no copy of the data. It is extremely cheap to copy and pass around.

std::string — Owner 🏠

  • Allocates memory, copies data
  • Expensive to initialize/copy
  • Can be modified freely
  • Always null-terminated
  • Safe — owns its data

std::string_view — Viewer 👁️

  • No allocation, no copy
  • Extremely cheap (just a pointer + length)
  • Read-only (cannot modify viewed string)
  • May NOT be null-terminated
  • Dangerous if source is destroyed
string_view_basic.cpp 
#include <iostream>
#include <string_view>

void printSV(std::string_view str) {       // ✅ Cheap! No copy made
    std::cout << str << '\n';
}

int main() {
    std::string_view s1 { "Hello, world!" };  // Views a C-style string literal
    printSV(s1);
    return 0;
}
✅ Best Practice
Prefer std::string_view over const std::string& for read-only string parameters. It works with C-style strings, std::string, and std::string_view arguments — no conversion needed.

Conversions Between String Types

ConversionAllowed?HowWhy
std::string → std::string_view ✅ Implicit Automatic Creating a view from an owner is cheap and safe
std::string_view → std::string ⚠️ Explicit only static_cast<std::string>(sv) Would make an expensive copy — must be intentional
C-style string → std::string_view ✅ Implicit Automatic C-style strings live for program lifetime, so safe to view
std::string_view → const char* ⚠️ Use .data() sv.data() Only safe if view is null-terminated (not always the case)

Constexpr Support

std::string_view fully supports constexpr, whereas std::string does not (in most compilers):

sv_constexpr.cpp 
constexpr std::string_view greeting { "Hello" };  // ✅ Valid!
// constexpr std::string name { "Alex" };            // ❌ Not valid in most contexts

The sv Suffix

sv_suffix.cpp 
using namespace std::string_view_literals;

auto sv = "Hello"sv;  // type: std::string_view
auto s  = "Hello"s;   // type: std::string
auto cs = "Hello";    // type: const char[]

⚠️ Dangling Views — The Primary Danger

⚠️ Critical Warning — Dangling string_view
Since std::string_view does NOT own its data, if the underlying string is destroyed or modified, the view becomes a dangling view. Accessing a dangling view is undefined behavior — it may crash, corrupt data, or appear to work (worst case!).

Dangerous Patterns:

dangling_views.cpp 
// ❌ DANGER 1: Viewing a temporary — destroyed at end of full-expression
std::string_view sv = std::string{"Hello"};  // std::string destroyed immediately!
std::cout << sv;  // UB: sv is dangling

// ❌ DANGER 2: Modifying the source string invalidates the view
std::string s { "Hello" };
std::string_view sv2 { s };
s += " World";              // s may reallocate — sv2 now points to freed memory!
std::cout << sv2;           // UB!

// ❌ DANGER 3: Returning a string_view to a local std::string
std::string_view getBad() {
    std::string local { "oops" };
    return local;  // ❌ local destroyed when function returns!
}

How to Fix a Dangling View:

fix_dangling.cpp 
std::string s { "Hello" };
std::string_view sv { s };
s += " World";   // s modified, sv is now dangling
sv = s;           // ✅ Reassign sv to point at the new, valid s
std::cout << sv;  // Now safe!

When Is It Safe to Return std::string_view?

✅ Safe to Return string_view When:
  1. You are returning a C-style string literal — these exist for the entire program lifetime.
  2. You are returning a std::string_view parameter that was passed in — the caller's string outlives the function.
⚠️ NEVER Return string_view When:
  • The view was created from a local std::string — it will be destroyed.
  • The view was created from a function's return value of type std::string (temporary).

Modifying the View (Shrinking the Window)

You can't modify the data through a string_view, but you can change what portion of the string is visible:

sv_modify_window.cpp 
std::string_view sv { "Hello, World!" };
sv.remove_prefix(7);   // Removes 7 chars from front → sv = "World!"
sv.remove_suffix(1);   // Removes 1 char from back  → sv = "World"
std::cout << sv;         // Outputs: World
⚠️ Warning — Not Null-Terminated!
After calling remove_prefix or remove_suffix, (and even before — any substring view) the string_view is NOT guaranteed to be null-terminated. Never pass a string_view to functions that expect a null-terminated C-string (e.g., legacy C functions) unless you are certain it is null-terminated.

📋 Sections 5.8–5.9 Summary

  • std::string_view is a cheap, read-only, non-owning view into existing string data.
  • It can view C-style strings, std::string, and other string_view data without making a copy.
  • string→string_view is implicit; string_view→string requires explicit cast.
  • A dangling view (pointing to destroyed data) causes undefined behavior.
  • Only safe to return string_view for C-style literals or passed-in views.
  • string_view is NOT guaranteed null-terminated — never assume it is.
  • Use remove_prefix()/remove_suffix() to adjust the visible window.

Question 1a: Why are named constants often a better choice than literal constants?

✅ Answer
  • Context: A name like maxStudents communicates intent; the raw literal 30 doesn't.
  • Maintainability: If the value needs to change, you update it in one place. With raw literals, you'd have to find and replace every occurrence — risking missing some.
  • No magic numbers: Named constants eliminate ambiguous hard-coded values in code.

Question 1b: Why are const/constexpr variables usually better than #define symbolic constants?

✅ Answer
  • No scoping: #define macros ignore C++ namespaces, classes, and functions — they pollute the entire file.
  • Debugger invisible: Debuggers see the substituted value, not the macro name, making debugging harder.
  • No type safety: Macros are untyped — the compiler cannot type-check them or warn about misuse.
  • Hard-to-understand errors: When a macro name collides or is misused, compiler errors reference the substituted text, not the macro name.

Question 2: Find 3 issues in this code:

buggy_code.cpp 
#include <cstdint>  // for std::uint8_t
#include <iostream>

int main() {
    std::cout << "How old are you?\n";
    std::uint8_t age{};
    std::cin >> age;
    std::cout << "Allowed to drive in Texas: ";
    if (age >= 16)
        std::cout << "Yes";
    else
        std::cout << "No";
    std::cout << '.' << '\n';
    return 0;
}
✅ Three Issues Found
  1. Issue 1 — uint8_t treated as char:
    std::uint8_t is typically an alias for unsigned char. When used with std::cin >>, the stream reads character input, not numeric input. If the user types 19, only the character '1' (ASCII value 49) is read. Since 49 ≥ 16, the code incorrectly reports "Yes" for a 1-year-old. Fix: Use int.
  2. Issue 2 — Magic number 16:
    The value 16 appears without context. Fix: Replace with constexpr int minDrivingAge { 16 };
  3. Issue 3 — Unsigned type for age:
    std::uint8_t is unsigned. If the user somehow inputs a negative number, it wraps around to a large positive value (≥ 16), and the program incorrectly reports "Yes." Fix: Use int, which handles negative inputs sanely.

Question 3a: Primary differences between std::string and std::string_view?

✅ Answer
  • std::string owns its data (dynamically allocated, deep copy). std::string_view is a non-owning viewer (just a pointer + length).
  • std::string is expensive to copy; std::string_view is extremely cheap.
  • std::string is always null-terminated; std::string_view is not guaranteed to be.
  • std::string is modifiable; std::string_view is read-only.
  • std::string_view fully supports constexpr; std::string does not.

Question 3b: What can go wrong with std::string_view?

✅ Answer

If the string being viewed is destroyed or reallocated (e.g., a local variable goes out of scope, or a std::string is modified causing reallocation), the string_view becomes a dangling view. Any access to a dangling view is undefined behavior — the program may crash, silently produce wrong results, or appear to work (unpredictably).

Question 4: Write a program that asks for the name and age of two people, then prints which is older.

✅ Full Solution
quiz_5x_q4.cpp 
#include <iostream>
#include <string>
#include <string_view>

std::string getName(int personNum) {
    std::cout << "Enter the name of person #" << personNum << ": ";
    std::string name{};
    std::getline(std::cin >> std::ws, name);  // Read full name with spaces
    return name;
}

int getAge(std::string_view name) {  // string_view: cheap, read-only
    std::cout << "Enter the age of " << name << ": ";
    int age{};
    std::cin >> age;
    return age;
}

void printOlder(
    std::string_view name1, int age1,
    std::string_view name2, int age2) {

    if (age1 > age2)
        std::cout << name1 << " (age " << age1 << ") is older than "
                  << name2 << " (age " << age2 << ").\n";
    else if (age2 > age1)
        std::cout << name2 << " (age " << age2 << ") is older than "
                  << name1 << " (age " << age1 << ").\n";
    else
        std::cout << name1 << " and " << name2 << " are the same age.\n";
}

int main() {
    std::string name1 { getName(1) };
    int         age1  { getAge(name1) };

    std::string name2 { getName(2) };
    int         age2  { getAge(name2) };

    printOlder(name1, age1, name2, age2);

    return 0;
}

Sample Output: David Jenkins (age 44) is older than John Bacon (age 37).

Question 5: Why can't age1 in main() be constexpr?

✅ Answer

age1 is initialized with the return value of getAge(), which internally reads from std::cin. User input is only available at runtime — the compiler cannot possibly know what the user will type while compiling the code. Since constexpr requires the initializer to be a compile-time constant expression, and getAge()'s return value is determined at runtime, age1 cannot be constexpr.

In this scenario, the appropriate keyword is const (if age1 shouldn't change after being read), or simply leave it as a regular int.

🗂️ Master Revision Summary — Chapter 5

Constants (5.1)

  • const = value cannot change after init
  • Must initialize at point of definition
  • Place const before the type
  • Prefer over #define macros always
  • Don't use on pass-by-value params or return-by-value

Literals (5.2)

  • Floating-point default = double (use f for float)
  • Use uppercase L, not l
  • "Hello" is 6 chars — includes '\0'
  • s suffix → std::string
  • sv suffix → std::string_view
  • Avoid magic numbers — use named constants

Numeral Systems (5.3)

  • Decimal (none), Hex (0x), Binary (0b), Octal (0)
  • Digit separator: ' (e.g., 1'000'000)
  • Use std::hex/std::oct/std::dec for output
  • Use std::bitset<N> for binary output
  • Manipulators are sticky — reset with std::dec

As-If Rule & Optimization (5.4)

  • Compiler can change code freely if observable behavior is same
  • Constant Folding: literal expressions computed at compile-time
  • Constant Propagation: constant variables replaced with values
  • Dead Code Elimination: unused code removed
  • const enables more aggressive optimization

Constant Expressions (5.5)

  • CE = fully evaluable at compile-time
  • const integral + CE init = CE variable
  • const double = NOT a CE (use constexpr)
  • Non-const vars = never CE
  • Runtime functions (cin, cout) = never CE

constexpr (5.6)

  • Guarantees compile-time evaluation
  • Works for ALL types (including double)
  • Initializer must be a constant expression
  • constexpr implies const
  • Use by default; fall back to const for runtime values

std::string (5.7)

  • Owner of data — dynamically allocated
  • Expensive to copy, flexible to use
  • Always use std::getline(cin >> std::ws, s)
  • Cast .length() with static_cast<int>
  • Never pass by value; returning by value OK

std::string_view (5.8–5.9)

  • Non-owning, read-only view (pointer + length)
  • Extremely cheap to copy
  • Use for read-only function parameters
  • Dangling view = undefined behavior
  • NOT null-terminated — never assume it is
  • Supports constexpr; string does not

⚡ Critical Rules to Never Forget

  • Always initialize const/constexpr variables at definition.
  • Floating-point literals are double — suffix with f for float.
  • Always use std::getline(std::cin >> std::ws, name) — never plain getline after cin >>.
  • string_view becomes dangling if the underlying string changes or is destroyed.
  • string_view is NOT null-terminated — don't assume it is.
  • const double is NOT a constant expression — use constexpr double.
  • Prefer constexpr > const > plain variable — use the most restrictive type possible.
